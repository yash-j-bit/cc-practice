# トランザクション安全性レポート

## 概要
在庫移動（Warehouse A → Warehouse B）のトランザクション安全性を検証した。

## テスト結果

| テスト | 結果 | 検証内容 |
|--------|------|----------|
| 移動失敗時のロールバック | ✅ Pass | 在庫不足で失敗した場合、出庫元の在庫が元に戻る |
| 成功時の在庫保存則 | ✅ Pass | 移動後、全倉庫合計が移動前と一致する |
| 複数回移動後の保存則 | ✅ Pass | 3回連続移動後も合計在庫が不変 |
| 受注失敗時のロールバック | ✅ Pass | 複数商品受注で1つが在庫不足の場合、全品の引き当てが巻き戻る |
| 連続移動の整合性 | ✅ Pass | A→B, B→A の連続移動後、各倉庫の在庫が正確に計算される |

## 検証詳細

### 1. ロールバック検証
```
初期状態: WH-A = 100, WH-B = 0
操作: WH-A → WH-B 200個移動（在庫不足）
結果: WH-A = 100（変化なし）, WH-B = 未作成
       stock_movements テーブルにも transfer レコードなし
```

### 2. 保存則検証（conservation of inventory）
```
操作前合計: 150（WH-A=100, WH-B=50）
操作: WH-A→WH-B 30個, WH-B→WH-A 10個, WH-A→WH-B 5個
操作後合計: 150（WH-A=75, WH-B=75）
```

### 3. 受注アトミック性
```
在庫: TX-1=100, TX-2=2 (WH-A)
受注: TX-1×5, TX-2×10 → InsufficientStockError
結果: TX-1=100（引き当て巻き戻し）, TX-2=2, orders=0件
```

## 実装方式
- `BEGIN` / `COMMIT` / `ROLLBACK` による明示的トランザクション管理
- `withTransaction` ヘルパー（`src/db/client.ts`）で自動 rollback
- エラー発生時は必ず `ROLLBACK` を実行してから re-throw

## 結論
libSQL のトランザクション機構により、在庫の整合性が保証されている。
中間状態でエラーが発生しても、データベースは操作前の状態に戻る（原子性 = Atomicity）。
